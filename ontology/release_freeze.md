# Core Hardening and Release Freeze v1.0

A release freeze stabilizes the seed root for copying, verification, and birth handoff. It is not a ceiling on future growth.

## Principle

The core can be released as a stable seed root when its required files, policies, contracts, eval suites, validation rules, and neutrality checks pass.

Release freeze means the v1.0 seed surface is stable. Future growth remains possible through new versions, approved migrations, and living memory outside the planted seed.

## Freeze boundary

The release freeze applies to the neutral seed repository. It does not include runtime code, product code, platform code, device code, provider code, or user interface code.

## Hardening requirements

A v1.0 release must confirm:

- seed identity exists
- doctrine exists
- policies exist
- contracts exist
- eval suites exist
- validation script covers required files
- blocked product and provider terms are absent
- local reasoning remains allowed without extra approval
- external providers remain governed
- guardian approval remains final for adoption
- runtime remains separate from seed
- growth remains open-ended with no artificial ceiling

## Release manifest

A release manifest records the version, file set, validation commands, eval commands, release status, and known limitations.

## Change after freeze

Changes after v1.0 must be treated as new version work or approved migration work. The v1.0 seed should remain reproducible and copyable.
